# 任务雷达：大规模并行协作

## 起源

一个 10+ 人的开发团队需要构建复杂的 DDD/CQRS 系统。他们不在同一个办公室。他们不能靠白板讨论、拍肩膀或每日站会来协调。

问题是：如何让 10+ 人同时在同一个代码仓库上工作而不互相阻塞？

答案：**任务雷达**——一个在 Miro.com 上的可视化协作看板，每个技术组件是一个任务节点，每个接口是一份合约，每个人并行工作，无需加锁。

## 运作方式

### 1. 架构拆解先行

在任何人写代码之前，团队共同按照洋葱架构模式拆解架构。产出：

- **层边界**（domain / application / infrastructure）及严格依赖规则
- **每个组件的接口定义**（Scala 中的 trait，在 Miro 上以方框图呈现）
- **Artifact 清单**——每个需要创建的文件完整列表，按层和组件组织

这个拆解过程在 Miro 上协作完成。结果是一幅可视化的"雷达屏"，每个节点都是一个等待认领的任务。

### 2. 雷达作为实时任务看板

在 Miro 看板上，每个 artifact 是一张卡片，包含：
- **名称和类型**（如 "DepositRule — InvariantRule"）
- **状态**（未开始 / 进行中 / 已完成 / 已阻塞）
- **负责人**（谁在做）
- **接口合约**（它必须实现的 trait 签名）
- **依赖**（它依赖哪些其他 artifact）

看板是**实时的**。每个人看到相同的状态。有人完成任务就更新卡片。有人开始任务就认领它。没人需要问"谁在做什么？"

### 3. 自主认领任务

没有任务分配会议。没有管理者分发工作。开发者自主：
- 扫描雷达上"未开始"的任务
- 选一个匹配自己技能或兴趣的
- 标记为"进行中"并写上自己的名字
- 完成后标记"已完成"，然后领下一个

这之所以有效，是因为**接口已经定义好了**。当你认领 `DepositRule` 时，你确切知道它需要什么签名（`(State, Money) => Either[Error, List[Event]]`）。你不需要问任何人这个函数应该长什么样。接口就是规格说明。

### 4. 无需加锁的并行

为什么这不会出现合并冲突或协调开销？

- **不同层 = 不同文件。** 写 `DepositRule`（domain）的人和写 `CreditBalanceBootstrap`（infrastructure）的人永远不会编辑同一个文件。
- **接口解耦了实现。** 实现 `CreditBalanceRepositoryImpl` 的人不需要等写 `MoneyTransferService` 的人完成——他们只需要 `CreditBalanceRepository` trait 被定义好。
- **任务之间无共享状态。** 每个 artifact 是独立的单元，有明确的输入和输出。唯一的共享理解是接口。

架构本身就是**协作协议**。代码结构就是团队的沟通媒介。

## 是什么让它成为可能

### 洋葱架构是前提

任务雷达在扁平架构中无法运作。如果每个文件都能导入其他任何文件，任务就不是独立的。洋葱架构严格的层边界创造了**自然的任务隔离**：

- Domain 任务不依赖任何东西（纯函数，零外部导入）
- Application 任务只依赖 domain 接口
- Infrastructure 任务依赖 application 接口

这个依赖图是一个**偏序关系**——它告诉你什么可以并行、什么必须顺序。大多数任务可以并行，因为它们位于依赖树的同一层级。

### 接口优先设计

每个组件在实现之前先暴露 trait。这不是编码偏好——这是**协作前提**。trait 是队友之间的合约。一旦接口稳定，实现可以由任何人并行进行。

### 可视化实时看板

Miro 提供了取代状态会议的共享可见性。看板回答了三个本来需要同步沟通的问题：
1. 有什么需要做？（所有"未开始"卡片）
2. 谁在做什么？（所有带负责人的"进行中"卡片）
3. 什么被阻塞了？（所有带依赖标记的"已阻塞"卡片）

## 经验教训

### 什么时候最有效

- **架构稳定时。** 接口的重大重构会破坏并行模型。
- **接口定义清晰时。** 签名模糊会导致实现不匹配时的返工。
- **团队理解领域时。** 任务雷达不会教你领域知识——它在你已经理解领域后帮你高效构建。

### 什么时候会失败

- **未发现的依赖。** 如果 artifact B 秘密地需要 artifact A 的实现细节（而不只是接口），做 B 的人会被阻塞。
- **接口频繁变更。** 如果 trait 定义一直改，所有对着它实现的人都要重做。
- **实现过程中才发现领域问题。** 当你在编码时发现领域模型不对，雷达需要在继续工作前更新。这要求整个团队重新同步。

### 元教训

架构**就是**团队的协调机制。好的架构使并行工作成为可能；差的架构强制串行工作。任务雷达让这一点可见——如果看板上有很多"已阻塞"卡片，问题出在架构，不是团队。

## 从人类团队到 Agent 团队

同样的原则适用于 AI Agent 并行开发：

- **接口 = Agent 任务规格说明。** 一个签名清晰的 trait 就是给 Agent 实现它的提示词。
- **雷达 = Agent 仪表盘。** 每张卡片变成一个 Agent 可以认领、执行和报告状态的任务。
- **架构 = 协调协议。** 如果架构定义了清晰的边界和接口，Agent 之间不需要互相通信。

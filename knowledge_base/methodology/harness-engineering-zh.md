# Harness 工程：文档作为记录系统

## 核心洞察

当 AI Agent 编写大部分代码时，文档不是给人读的——是给 Agent 做推理用的。代码仓库成为项目所有知识的**记录系统**。

Agent 从仓库中无法发现的东西，就是不存在的东西。

这意味着：Slack 讨论、会议记录、Google Docs、人的隐性知识——对 Agent 来说**都不可见**。只有仓库中版本化的文件才是真实的。

## 三层知识体系

本项目将知识组织为三个层次，各有明确分工：

```
第 1 层：CLAUDE.md           →  规则（"地图"）
第 2 层：knowledge_base/      →  参考（"地形"）
第 3 层：knowledge_base/prompts/ →  指令（"任务简报"）
```

### 第 1 层：规则 — `CLAUDE.md`（约 70 行）

**地图**，始终加载在上下文中。它告诉 Agent：

- 技术栈和版本号（避免猜错）
- 架构铁律（内层绝不导入外层）
- 编码约定（用 Either 表达错误、禁止 null、不可变数据）
- Protobuf "三件套"规则（三个文件必须同步修改）
- TDD 陷阱（`.thenStop()` 不能用 `runCommand`）
- 目录地图（哪里找什么）

**原则：** 足够短以始终在上下文窗口中。充当目录，而非百科全书。Agent 需要细节时，顺着指针去读。

### 第 2 层：参考 — `knowledge_base/`

**地形**，按需加载。分为：

- `architecture/` — 每一层是什么、包含什么、能导入什么
- `artifacts/` — 30 个文件，每个定义一个代码 artifact 类型，含文件路径模式、约束、参考模板
- `saga_framework/` — Saga 引擎文档，含使用指南和架构图

**原则：** 渐进式披露。Agent 从 CLAUDE.md 出发，按链接追踪到相关参考文件。它读的是 1 个文件，不是 40 个。

### 第 3 层：指令 — `knowledge_base/prompts/`

**任务简报**。预写的提示词，精确告诉 Agent 每种 artifact 类型该产出什么。这些提示词的结构确保 Agent 收到：上下文、约束、期望的输出格式。

**原则：** 可复用。一个提示词一旦验证有效，对该 artifact 类型就永远有效。

## 地图，而非百科全书

反面模式是搞一个巨大的 AGENTS.md 文件。为什么它不行：

1. **上下文稀缺。** 巨大的指令文件挤掉了代码、任务和相关文档。
2. **什么都重要 = 什么都不重要。** Agent 变成了局部模式匹配，而非有意识的导航。
3. **立即腐烂。** 庞杂的手册会成为过时规则的坟场。
4. **无法验证。** 单个 blob 无法进行覆盖率、新鲜度、所有权的机械检查。

替代方案：`CLAUDE.md` 指向更深的信息源。每个参考文件有唯一的明确职责。交叉链接连接相关主题。

## 保持文档新鲜度

文档如果不验证就会衰减。理想状态（本项目尚未完全实现）：

- Linter 规则检查交叉链接是否有效
- CI 作业验证架构约束（无 domain→infra 导入）
- 定时运行的 "doc-gardening" Agent 扫描过时文档并发起修复 PR

没有机械强制，文档就会与代码偏离。有了它，系统保持自洽。

## 参考：Harness Engineering 原则

这些原则源自 OpenAI 的 Codex 驱动开发实验：

1. **代码仓库是记录系统。** 仓库之外的知识对 Agent 不存在。
2. **渐进式披露。** 从地图开始，不搞百科全书。指向细节，不嵌入细节。
3. **机械强制。** Markdown 里的规则是愿望。Linter/CI 里的规则是事实。
4. **Agent 需要约束，不需要指令。** 宽边界 + 局部自治胜过微观管理。
5. **定期垃圾回收。** 坏模式会蔓延。早清除、常清除。

# Agent 并行开发：下一次进化

## 愿景

任务雷达方法论是为人类团队设计的。下一次进化是让它适用于**人机混合团队**——AI Agent 从雷达认领任务、实现、运行测试、报告状态，而人类充当架构师、审查者和决策者。

**人类掌舵。Agent 执行。**

## 三阶段演进

### 阶段 1：人类团队（当前）

```
Miro 任务雷达  ←→  人类开发者
```

- 架构被拆解为 artifact
- 接口协作定义
- 人类认领任务、实现、测试、提交
- 任务雷达显示实时状态

这一阶段已经**验证**——它成功支撑了 10+ 人构建本项目。

### 阶段 2：Agent 辅助（近期）

```
Miro 任务雷达  ←→  人类架构师
               ←→  AI Agent（实现者）
```

- 人类架构师定义接口和 artifact 模板
- AI Agent 从雷达认领任务
- 每个 Agent：读取接口 → 按模板实现 → 写测试 → 发起 PR
- 人类审查 PR（但越来越信任 Agent 间互相审查）
- 任务雷达实时显示 Agent 进度

这一阶段的关键使能器是 `knowledge_base/artifacts/` 目录。每个 artifact 模板就是给 Agent 的**提示词规格说明**：

```
Artifact: InvariantRule
接口:     (State, Param) => Either[Error, List[Event]]
模板:     knowledge_base/artifacts/domain_invariant.md
测试:     用纯输入/输出断言验证
```

当 Agent 认领"实现 WithdrawRule"时，它读取模板，理解期望的形状，产出符合规范的代码。模板就是指令。

### 阶段 3：自主 Agent 团队（未来）

```
任务雷达  ←→  人类架构师（仅设计）
          ←→  AI Agent（全生命周期）
          ←→  Agent 审查者
          ←→  文档园艺 Agent
```

- Agent 处理完整开发生命周期：设计 → 实现 → 测试 → 审查 → 合并
- 人类聚焦：领域发现、架构演进、质量标准、边缘案例识别
- 多个 Agent 并行工作，各自处理不同 artifact
- "元 Agent" 编排：分配任务、解决冲突、卡住时升级给人类
- 任务雷达成为人类**观察自主 Agent 工作的可见层**

## 代码生成流水线

使这一切成为可能的技术基座：

```
DDD DSL 元数据
    ↓
Claude Code + Skills + MCP
    ↓
生成代码骨架（目录结构、接口、protobuf）
    ↓
Claude Code 填空实现（遵循 artifact 模板）
    ↓
TDD 7 步提交循环
    ↓
Agent 自审查 → PR → 合并
```

### DDD DSL 元数据

用领域特定语言捕获领域模型：实体、值对象、聚合、事件、命令、Saga。这些元数据驱动初始代码生成，产出具有正确目录结构和接口定义的骨架。

### Claude Code + Skills + MCP

Skills 提供领域特定能力（如"生成一个 InvariantRule"、"添加 Saga 参与者"）。MCP（Model Context Protocol）连接 Claude 与外部工具：文件系统、git、构建系统、测试运行器。

### TDD 7 步提交循环

每个 artifact 实现，Agent 遵循纪律性提交序列：

1. **写失败测试**——定义期望行为
2. **提交测试**——`test: add failing test for [artifact]`
3. **运行测试**——验证失败（Red）
4. **实现**——写最小代码通过测试
5. **运行测试**——验证通过（Green）
6. **重构**——保持测试绿色前提下清理代码
7. **提交实现**——`feat: implement [artifact]`

每次提交是一个可检查的进度单位。如果出错，Agent 可以回退到最后已知良好状态。对人类观察者来说，提交历史讲述了构建了什么以及如何构建的故事。

## 人类角色

在这个模型中，人类**不再是文件编辑者**，而成为：

| 角色 | 职责 |
|------|------|
| **领域专家** | 定义系统应该做什么，澄清业务规则 |
| **架构师** | 设计接口，执行边界，演进结构 |
| **质量门** | 审查关键 PR，定义验收标准，发现边缘案例 |
| **工具构建者** | 创建 skills、模板、linter 和提示词，让 Agent 更有效 |
| **排障者** | 当 Agent 卡住时介入调查，识别缺失的约束或工具 |

## Agent 角色

Agent 接管**执行**层：

| 角色 | 职责 |
|------|------|
| **实现者** | 按接口和模板编写代码 |
| **测试者** | 按 TDD 纪律编写和运行测试 |
| **审查者** | 按架构约束审查其他 Agent 的 PR |
| **园艺师** | 检测和修复代码漂移、过时文档、质量退化 |
| **状态报告者** | 在任务雷达上更新进度、阻塞和完成状态 |

## 为什么这个项目是参考标准

本项目（`playAkkaCQRS`）作为代码生成流水线的**金标参考实现**，因为：

1. **它是用流水线要生成的精确架构构建的。** 洋葱 + DDD + CQRS/ES + Saga。
2. **每个 artifact 类型都有模板**在 `knowledge_base/artifacts/`——这些是代码生成的提示词。
3. **编码约定被机械化了**——FP 模式、DSL、类型安全规则嵌入架构，而非仅文档化。
4. **它展示了全栈**——从领域实体到基础设施持久化，从 Web UI 到 WebSocket 流，从正常路径到边缘案例恢复。

当流水线生成代码时，输出应该看起来像这个项目。当 Agent 实现一个功能时，质量标准是"它是否符合 playAkkaCQRS 中的模式？"

## 还需要构建的

这一愿景需要尚未就位的基础设施：

- **Agent 可访问的任务雷达。** Miro 看板需要机器可读 API，让 Agent 可以查询任务状态。
- **机械约束执行。** Linter 在 CI 时验证层依赖、protobuf 同步和编码约定。
- **Agent 自审查能力。** 让 Agent 运行 `sbt test`、检查覆盖率、在发起 PR 前验证自己工作的脚本。
- **多 Agent 编排。** 一个协调器分配不冲突的任务给 Agent、检测 Agent 卡住并升级。
- **人类可见仪表盘。** 任务雷达作为实时仪表盘展示 Agent 进度，而非仅人类进度。

这些是从阶段 2 迈向阶段 3 需要解决的工程挑战。
